# Phase completion log (moved out of docs/TODO.md on 2026-07-04)

These are the completion narratives that previously lived inline in the master TODO. They are history, not work, so they live here. One-line summaries of the same work are in CHANGELOG.md under Unreleased.

## Phase 1 — prototype port slices

- Slice 1, Shell: AppShellView/AppSidebar/AppNavigation landed in Features/App. ActiveTab carries the new IA. The delegate-owned window is hidden-title/transparent/no-zoom, width fixed at 820 with vertical resize, and the rail collapse hides the real minimize button.
- Slice 2, Configuration: ConfigurationPane wired to real state. Appearance persists via `appearancePreference` and drives the window scheme, launch-on-login uses the SMAppService wrapper, Sparkle check/auto-update toggles bind to the real updater, permissions pills read live AppStore state, retention/clipboard/duration dropdowns bind to typed settings enums, and "Simulate keypresses" inverts `useClipboardPaste`. Codex split the file into pane/controls/recording-surface-mocks. New Core settings fields shipped; the settings schema moved to its own file for the 500-line rule. Prototype pane deleted.
- Slice 3, Sound: SoundPane binds mic auto-raise, silence removal, dynamic normalization, playback behavior, effects style, and volume to real settings. The microphone menu lists real input devices via RecordingClientLive with System Default mapping. SoundEffectsService plays the Default/Classic asset sets and honors .off and volume. Prototype pane deleted.
- Slice 4, Modes: Features/Modes/{ModesPane, ModeDraft, ModesComponents, ModesModelPicker}. The Default mode reads and writes real settings (preset ↔ textTransformMode including the new Note preset, language ↔ outputLanguage, voice model ↔ selectedModel from TranscriptionModelCatalog, LLM ↔ textTransformModel from CloudLanguageModels, playback ↔ recordingAudioBehavior, system audio ↔ recordingInputMode, auto paste ↔ autoPasteResult). Extra modes stay mock until the Core Mode model. Prototype pane deleted.
- Slice 5, Model library: ModelLibraryPane renders the real adapter sections (local dictation, cloud dictation, streaming preview, cloud text, support assets) with real downloads, progress, deletes, sizes, and source-backed metrics; selection binds to selectedModel/alwaysOnStreamingModel/textTransformModel. No fake scores. The shared download control was promoted to UI/TLModelDownloadControl and ModelDownloadStore's selection layer split into its own extension file. Experimental-models filtering was skipped honestly — the adapter exposes no such flag yet. Prototype pane deleted.
- Slice 6, History: HistoryPane runs on live TranscriptRecord data — FTS5 search, day groups from real dates, app filters built from actual records, persisted title edits, Raw/Processed tabs on rawText/finalText, real AVAudioPlayer play/stop on the recorded audio. Old History UI deleted. Scrubbing is a follow-up (in the TODO).
- Slice 7, Home: HomePane computes the stats strip from real TranscriptRecords (words, WPM, distinct apps, time saved against a named typingWPM constant), reads Today from the store with real deep-link ids, and reuses the History app-icon mapping. HistoryItem.samples deleted.
- Slice 8, Hot Mic + License: HotMicPane binds Enable to alwaysOnEnabled, seeds Paste/Dump recorders from the real always-on hotkeys, and uses the real mic menu. LicensePane stays visually mock with constants extracted, pending the Phase 4 transport. Both prototype panes deleted.
- Prefix decision: `Proto*` became `TL*` and the kit moved to TimberVox/UI/ (plain names like Section/Divider collide with SwiftUI). The prototype shell is now a preview wrapper around the real AppShellView.
- Dead-code sweep (grep-verified): old General/About/Recording/Shortcuts panes, the old models UI, and HotKeyView deleted. Kept: ModelLibraryAdapter (live), SettingsStyles (used by WordRemappingsView), and WordRemappingsView itself — the only editor for word removals/remappings, currently unmounted; re-homing is in the TODO.

## Phase 2 — data + persistence (landed portion)

- GRDB transcript store core layer: GRDB 7.11.1 pinned in TimberVoxCore; TranscriptStore + TranscriptRecord with the recording table, synchronized FTS5 search over title/raw/final text, retention sweep, idempotent legacy import, and unit tests. Decision settled: GRDB over Core Data (inspectable schema, exact migrations, FTS5, live DB assertions). Superwhisper's own storage validates the architecture — SQLite with GRDB migration tables, a `recording` table, an FTS5 table for transcript search, and per-recording folders holding audio plus meta.json. Ours is modeled the same way: searchable metadata and text in SQLite, file-backed artifacts in per-recording folders.
- Retention: the startup sweep enforces recordingRetention against the TranscriptStore, mirrors removals out of the JSON history, and deletes the audio files; covered by TranscriptStoreRuntimeTests.
- Runtime wiring: ServiceContainer owns the store at Application Support/com.chiejimofor.timbervox/transcripts.sqlite with startup legacy import and sweep; TranscriptionStore dual-writes TranscriptRecord; History and Home read the store.
